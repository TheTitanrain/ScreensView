# Adaptive Tile Grid ŌĆö Design Spec

**Date:** 2026-03-31
**Status:** Draft

## Problem

`MainWindow` displays computer screenshot tiles in a `WrapPanel` with hardcoded `Width="320" Height="210"`. When the window width does not perfectly fit an integer multiple of 320 px, empty space appears on the right side of the grid.

## Goal

Make the tile grid fill the full available width at all window sizes with no leftover space. The number of columns and tile dimensions must update automatically whenever the window is resized.

## Behavior

- Tiles fill the row uniformly ŌĆö no empty horizontal space.
- The number of columns is computed from a minimum tile width of **240 px**: `columns = Math.Max(1, (int)(availableWidth / 240))`.
- Each tile's width is `availableWidth / columns` (exact, no remainder).
- Each tile's height scales proportionally to preserve the original **320 ├Ś 210** aspect ratio (`height = width ├Ś 210 / 320`).
- Tile spacing (margin) is preserved inside the `ItemWidth` slot.
- No manual size control ŌĆö fully automatic on resize.

## Implementation

### XAML changes (`MainWindow.xaml`) ŌĆö 3 changes

1. Add `x:Name="_screensScrollViewer"` and `SizeChanged="ScrollViewer_SizeChanged"` to the `ScrollViewer`.
   `_screensScrollViewer` is auto-generated by WPF from `x:Name` ŌĆö no manual field declaration needed.
2. Add `Loaded="WrapPanel_Loaded"` to the `WrapPanel` inside `ItemsPanelTemplate`.
   `x:Name` on elements inside `ItemsPanelTemplate` is not reachable from code-behind (template scope); a `Loaded` handler is the correct way to capture the panel instance.
3. Remove `Width="320" Height="210"` from the tile `Border` in the `DataTemplate`.
   `WrapPanel.ItemWidth` / `ItemHeight` will control sizing uniformly instead.

### Code-behind (`MainWindow.xaml.cs`)

Add `using System.Windows.Controls;` at the top ŌĆö not currently imported (existing code uses the fully-qualified name `System.Windows.Controls.Border` on the cast at line 50 precisely because the namespace is absent). After adding the `using`, that fully-qualified cast can be simplified to just `Border`.

```csharp
private const double MinTileWidth = 240.0;
private const double TileAspectRatio = 210.0 / 320.0; // 210 / 320

private WrapPanel? _tileWrapPanel;

private void WrapPanel_Loaded(object sender, RoutedEventArgs e)
{
    _tileWrapPanel = (WrapPanel)sender;
    // WrapPanel.Loaded fires as part of the initial layout pass (children before
    // parents), so _screensScrollViewer already has a valid ViewportWidth here.
    UpdateTileSize();
}

private void ScrollViewer_SizeChanged(object sender, SizeChangedEventArgs e)
{
    UpdateTileSize();
}

private void UpdateTileSize()
{
    if (_tileWrapPanel == null) return;
    // Use ViewportWidth (not ActualWidth) to exclude the vertical scrollbar width,
    // which would otherwise cause tiles to overflow or trigger a horizontal scrollbar.
    double availableWidth = _screensScrollViewer?.ViewportWidth ?? 0;
    if (availableWidth <= 0) return;
    int columns = Math.Max(1, (int)(availableWidth / MinTileWidth));
    double tileWidth = availableWidth / columns;
    _tileWrapPanel.ItemWidth = tileWidth;
    _tileWrapPanel.ItemHeight = tileWidth * TileAspectRatio;
}
```

**Initial sizing:** `WrapPanel_Loaded` calls `UpdateTileSize()` to set correct dimensions on first render. By the time any `Loaded` event fires, WPF has already completed the measure/arrange layout pass, so `_screensScrollViewer.ViewportWidth` is already valid. The `_tileWrapPanel == null` guard and the `availableWidth <= 0` guard make `UpdateTileSize` safe to call from any context.

## Out of Scope

- Manual tile size slider
- Persisting tile size preference
- Changes to any other window
