# Визуальное обновление главного окна Viewer — дизайн-документ

**Дата:** 2026-03-31
**Проект:** ScreensView.Viewer (WPF, net8.0-windows)

---

## Контекст

Текущее главное окно `ScreensView.Viewer` функционально рабочее, но визуально выглядит как базовый WPF-экран:

- системный `ToolBar` без выраженной иерархии действий
- тёмные карточки с жёсткими границами и минимальной типографикой
- белый пустой фон, из-за которого интерфейс ощущается незавершённым
- общее ощущение, что элементы собраны из разных визуальных паттернов

Пользователь хочет сделать внешний вид более красивым и современным, но без тяжёлого тёмного интерфейса и без радикальной смены текущей структуры окна.

Во время согласования выбрано направление:

- общий стиль: `современный dashboard`
- компоновка: сохранить текущий каркас окна, без боковой навигации и крупных hero-блоков
- шапка: мягкий цветовой акцент, без агрессивной или перегруженной панели
- масштаб задачи: обновить главное окно через общую систему стилей, но не делать полный redesign всех диалогов в этом проходе

## Дизайн

### Подход

Используется вариант `главное окно + базовая система стилей`.

Это означает:

- визуальные ресурсы и базовые стили выносятся в `App.xaml`
- `MainWindow.xaml` пересобирается на этих ресурсах
- существующая логика окна сохраняется
- вспомогательные окна (`AboutWindow`, `ComputersManagerWindow`, `CredentialsDialog` и т.д.) не перерабатываются целиком, но смогут использовать те же ресурсы позднее

Подход выбран как баланс между качеством результата и объёмом изменений:

- лучше, чем локальная перекраска только `MainWindow.xaml`
- заметно безопаснее и уже по объёму, чем полный refresh всей viewer-части

### Общий визуальный стиль

Главное окно должно получить аккуратный современный вид в светлой палитре:

- фон окна не чисто белый, а светлый холодный (`slate` / `blue-gray`)
- карточки экранов светлые, с увеличенным радиусом, тонкой рамкой и мягкой тенью
- шапка оформлена как `command bar`, а не как стандартный WPF `ToolBar`
- основной акцент остаётся на карточках экранов и статусах
- декоративность умеренная: без тяжёлых градиентов на всём окне, без “неонового” dashboard-эффекта

### Структура главного окна

`MainWindow.xaml` делится на два читаемых визуальных блока:

1. **Header / command bar**
2. **Screens grid**

#### 1. Header / command bar

Верхняя часть окна заменяет текущий системный `ToolBar` на кастомную панель с мягким акцентом.

Блок с названием приложения и маркетинговым подзаголовком не обязателен: command bar может быть собран только из управляющих элементов, если это даёт более компактный и рабочий вид.

Содержимое сохраняется по смыслу:

- управление списком компьютеров
- настройка интервала обновления
- запуск/остановка polling
- обновление агентов
- автозапуск
- переход в окно «О программе»

Но подача меняется:

- кнопки получают единый стиль
- основное действие (`Запустить` / `Остановить`) визуально выделяется
- блок интервала выглядит как цельный контрол, а не набор разрозненных элементов
- шапка выравнивается по вертикали и горизонтали с предсказуемыми отступами
- приоритет в шапке отдаётся действиям и настройкам, а не branding-элементам

Шапка должна выглядеть выразительнее текущей, но оставаться спокойной:

- допустим мягкий градиент или оттеночный фон внутри панели
- недопустимо превращать шапку в самый тяжёлый визуальный элемент окна

#### 2. Screens grid

Сетка скриншотов остаётся центральной частью окна и продолжает использовать `ItemsControl` + `WrapPanel`.

Меняются:

- внешний вид карточки
- внутренние отступы
- нижний статусный блок
- заглушки для отсутствующего скриншота и блокировки

Карточка должна выглядеть как современный мониторинговый tile:

- светлая оболочка
- отдельная рамка под изображение
- аккуратный footer со статусом, именем и временем
- более чистая визуальная иерархия

Размеры карточек допускается слегка скорректировать, если это улучшает плотность сетки и читаемость, но без перелома текущей логики размещения.

### Состояния карточки

Существующие состояния сохраняются:

- `Unknown`
- `Online`
- `Offline`
- `Error`
- `Locked`
- отсутствие скриншота

Изменяется только визуальная подача:

- статусный индикатор становится аккуратнее и контрастнее
- footer получает более читаемый текст и визуальный баланс
- placeholder “Нет изображения” выглядит как осознанная пустая карточка, а не системная заглушка
- `Locked` остаётся с overlay, но становится визуально чище и лучше вписывается в общий стиль

Новая бизнес-логика не добавляется:

- двойной клик по карточке работает как раньше
- для `Locked` zoom-окно по-прежнему не открывается
- правила смены `Status` не меняются

### Общие ресурсы и стили

В `App.xaml` добавляются общие ресурсы, которые затем используются минимум в `MainWindow.xaml`:

- кисти палитры
- базовые толщины границ
- размеры скругления
- стили кнопок
- стиль акцентной кнопки
- стиль toggle-кнопки запуска/остановки
- стиль `CheckBox`
- стиль `Slider`
- стиль карточки компьютера
- стиль для вспомогательных текстов и small captions

Цель:

- убрать дублирование локальных стилей в окне
- обеспечить цельность интерфейса
- подготовить ресурсную базу для дальнейшего постепенного обновления других окон

### Изменения во ViewModel

`MainViewModel` получает только presentation-свойства, нужные для summary row и упрощения XAML:

- количество активных компьютеров
- количество `Online`
- количество проблемных компьютеров (`Offline` и `Error`)
- текст текущего режима polling / интервала

Свойства не меняют логику приложения и рассчитываются на основе уже существующих данных.

`ComputerViewModel` при необходимости получает 1–2 presentation-свойства, если это заметно упрощает XAML:

- например, текст статуса
- или признак наличия скриншота

Если те же эффекты можно получить простыми привязками и триггерами без разрастания разметки, новые свойства можно не добавлять.

### Что не входит в задачу

В рамках этого обновления не добавляются:

- фильтры по компьютерам
- поиск
- сортировка/группировка
- боковая навигация
- отдельный экран аналитики
- redesign всех вспомогательных окон viewer-части
- новая логика обновления или мониторинга

Это принципиально задача про визуальное обновление главного окна, а не про расширение функциональности.

## Проверка

1. `dotnet build ScreensView.Viewer` проходит без ошибок XAML и привязок ресурсов.
2. `dotnet test` проходит без регрессий.
3. Главное окно визуально сохраняет текущие сценарии:
   - управление компьютерами открывается
   - polling запускается и останавливается
   - интервал обновления меняется
   - обновление агентов запускается
   - автозапуск переключается
   - окно «О программе» открывается
4. Карточки корректно отображают состояния `Online`, `Offline`, `Error`, `Locked`, отсутствие скриншота.
5. Двойной клик по доступной карточке по-прежнему открывает `ScreenshotZoomWindow`.
6. `Locked`-карточка по двойному клику по-прежнему не открывает zoom.

## Примечание по реализации

Приоритет реализации:

1. сначала ввести общие ресурсы и стили
2. затем собрать на них новое `MainWindow.xaml`
3. затем добавить минимально необходимые presentation-свойства
4. только после этого корректировать тесты и документацию

Это снижает риск того, что визуальный рефакторинг приведёт к бессистемному росту XAML или к смешиванию UI-изменений с логическими изменениями.
