# Зашифрованный внешний файл подключений Viewer — дизайн-документ

**Дата:** 2026-03-30
**Проект:** ScreensView.Viewer (WPF, net8.0-windows)

---

## Контекст

Сейчас Viewer уже разделяет два типа данных:

- локальные настройки программы в `%AppData%\ScreensView\viewer-settings.json`
- список подключений в `%AppData%\ScreensView\computers.json`

Но путь к файлу подключений зашит в код. Из-за этого несколько пользователей не могут использовать один и тот же набор подключений. Пользователю нужна возможность выбрать внешний файл подключений через UI и сохранить этот выбор локально для своей копии Viewer.

Дополнительное требование: внешний файл должен содержать `ApiKey` каждого хоста, но не в открытом виде. Для этого весь внешний файл подключений должен быть зашифрован паролем. Каждый пользователь может сохранить этот пароль локально на своём компьютере, чтобы не вводить его при каждом запуске Viewer.

Также пользователю нужен явный путь назад к локальному `%AppData%\ScreensView\computers.json`.

---

## Дизайн

### Разделение данных

`viewer-settings.json` остаётся локальным файлом настроек Viewer. Он продолжает хранить индивидуальные настройки программы и дополнительно получает новые поля:

- `ConnectionsFilePath`
- `ConnectionsFilePasswordEncrypted`

`ConnectionsFilePath` задаёт путь к внешнему файлу подключений.

`ConnectionsFilePasswordEncrypted` хранит пароль к внешнему файлу не в открытом виде, а в DPAPI-шифрованном виде для текущего Windows-пользователя.

Если `ConnectionsFilePath` пустой, активным источником является локальный `%AppData%\ScreensView\computers.json`.

Если `ConnectionsFilePath` задан, активным источником является внешний зашифрованный файл подключений.

### Локальный файл подключений

Локальный `%AppData%\ScreensView\computers.json` остаётся в существующем формате:

- список `ComputerConfig`
- `ApiKey` хранится как `ApiKeyEncrypted` через DPAPI

Это поведение не меняется и используется в локальном режиме.

### Внешний файл подключений

Внешний файл подключений хранит не открытый JSON, а зашифрованный контейнер.

Внутри контейнера шифруется весь JSON списка подключений целиком, а не отдельные поля. Это упрощает модель и защищает не только `ApiKey`, но и весь набор хостов.

Минимальная структура контейнера:

- `Version`
- `KdfSalt`
- `Nonce`
- `Tag`
- `Ciphertext`

Все бинарные значения хранятся в Base64.

Рекомендуемая криптография:

- derivation key from password: `PBKDF2-SHA256`
- file encryption: `AES-GCM`

Это даёт совместимость с `net8.0-windows` и достаточную защиту для файла, который может лежать в общей папке.

### Работа с паролем

Для подключения внешнего файла Viewer использует отдельный диалог пароля.

Сценарии:

- при создании нового внешнего файла пользователь вводит пароль, подтверждение пароля и может отметить `Запомнить пароль на этом компьютере`
- при открытии существующего внешнего файла Viewer сначала пытается использовать локально сохранённый пароль
- если сохранённого пароля нет или он не подходит, Viewer показывает диалог ввода пароля
- после успешного открытия Viewer снова предлагает сохранить пароль локально, если он ещё не сохранён

Если пользователь выбирает запомнить пароль, он сохраняется только локально через DPAPI в `ConnectionsFilePasswordEncrypted`.

Если сохранённый пароль перестал подходить, Viewer очищает его и снова просит ввести пароль вручную.

### Startup flow

Открытие внешнего файла больше не может происходить только внутри `MainWindow`, потому что сначала нужно понять активный источник подключений и, при необходимости, получить пароль.

Поэтому запуск Viewer меняется так:

1. `App` читает `viewer-settings.json`.
2. Если `ConnectionsFilePath` пустой, используется локальный `computers.json`.
3. Если `ConnectionsFilePath` задан, Viewer пытается открыть внешний файл через сохранённый пароль.
4. Если пароль отсутствует или не подходит, Viewer показывает диалог ввода пароля.
5. Только после успешного разрешения источника подключений создаётся `MainWindow` и `MainViewModel`.

Если внешний файл не удаётся открыть автоматически, не должно быть тихого fallback на локальный файл. Пользователь должен явно выбрать дальнейшее действие.

Рекомендуемое поведение при ошибке открытия внешнего файла на старте:

- показать сообщение об ошибке
- предложить повторить ввод пароля
- либо явно переключиться на локальный файл подключений
- либо отменить запуск

### Поведение UI

Управление источником подключений добавляется в окно `ComputersManagerWindow`, а не в общее окно настроек, потому что пользователь работает именно со списком компьютеров.

В окно добавляются:

- кнопка `Файл подключений...`
- текст текущего источника: локальный файл или путь к выбранному внешнему файлу
- действие `Использовать локальный файл подключений`

### Предупреждение о безопасности

Перед созданием или подключением внешнего файла Viewer показывает предупреждение:

`Файл подключений будет доступен всем, у кого есть сам файл и пароль к нему. Храните его только в папке с ограниченным доступом.`

Без подтверждения пользователя переключение не выполняется.

### Переключение на внешний файл

Переключение применяется сразу, без перезапуска приложения.

Алгоритм:

1. Пользователь выбирает путь к внешнему файлу.
2. Viewer показывает предупреждение о безопасности.
3. Если файл не существует, Viewer запрашивает новый пароль, создаёт зашифрованный файл и переносит в него текущий список подключений.
4. Если файл существует, Viewer пытается открыть его сохранённым локально паролем или запрашивает пароль у пользователя.
5. После успешного открытия Viewer сохраняет `ConnectionsFilePath` в `viewer-settings.json`.
6. Если пользователь выбрал запомнить пароль, Viewer сохраняет DPAPI-шифрованный пароль в `ConnectionsFilePasswordEncrypted`.
7. Список компьютеров в UI обновляется сразу.

### Возврат к локальному файлу

Возврат к локальному хранению выполняется явным действием пользователя.

Алгоритм:

1. Пользователь выбирает `Использовать локальный файл подключений`.
2. Viewer переключается обратно на `%AppData%\ScreensView\computers.json`.
3. Если локальный файл уже существует, он загружается.
4. Если локального файла ещё нет, он создаётся из текущего набора подключений.
5. `ConnectionsFilePath` очищается в `viewer-settings.json`.
6. `ConnectionsFilePasswordEncrypted` очищается, чтобы локально не оставался ненужный пароль.

Автоматический fallback на локальный файл без явного согласия пользователя не выполняется.

### Архитектура сервисов

Чтобы не смешивать два разных формата хранения в одном классе, хранилища лучше разделить.

Рекомендуемая структура:

- `ComputerStorageService` остаётся сервисом локального `%AppData%\ScreensView\computers.json`
- добавляется `EncryptedComputerStorageService` для внешнего файла, зашифрованного паролем
- добавляется координирующий сервис или фабрика активного источника подключений, которая:
  - читает `viewer-settings.json`
  - выбирает локальное или внешнее хранилище
  - запрашивает пароль при необходимости
  - выполняет переключение источника

`MainViewModel` не должен знать детали криптографии. Он должен работать с активным хранилищем подключений и уметь обновить коллекцию `Computers`, когда пользователь переключил источник.

### Поведение `MainViewModel`

`MainViewModel` получает явную операцию смены источника подключений.

Она должна:

- принять уже готовое активное хранилище подключений
- при переключении либо загрузить подключения из нового источника, либо сначала экспортировать в него текущий список
- заменить содержимое коллекции `Computers` без пересоздания всего окна
- при активном polling перезапустить `ScreenshotPollerService` на новом наборе компьютеров
- не менять текущее состояние, если переключение завершилось ошибкой

### Обработка ошибок

Если внешний файл недоступен, пароль неверный, файл повреждён или запись завершилась ошибкой:

- Viewer показывает пользователю явное сообщение об ошибке
- текущий источник подключений остаётся прежним
- `ConnectionsFilePath` и `ConnectionsFilePasswordEncrypted` не должны частично обновляться
- список компьютеров в UI не меняется

### Тесты и документация

Изменение должно идти через TDD для новой логики хранения и переключения источника подключений. Минимум: новые тесты для зашифрованного внешнего файла, обновлённые тесты для локального режима и тесты на переключение источника без потери текущего состояния.

Пользовательская документация в `README.md` обновляется обязательно. В ней должны быть описаны:

- локальный режим хранения
- внешний зашифрованный файл подключений
- ввод и локальное сохранение пароля
- явный возврат к локальному файлу подключений

## Проверка

1. `dotnet test` проходит без ошибок, включая новые и обновлённые тесты для storage, startup flow и переключения источника.
2. По умолчанию Viewer продолжает работать через `%AppData%\ScreensView\computers.json`.
3. Пользователь может выбрать внешний файл подключений через UI.
4. Новый внешний файл создаётся только в зашифрованном виде.
5. Существующий внешний файл открывается только при корректном пароле.
6. При выборе `Запомнить пароль на этом компьютере` пароль сохраняется локально через DPAPI и больше не требуется на следующем запуске для этого пользователя.
7. Если сохранённый пароль больше не подходит, Viewer очищает его и просит пароль заново.
8. Пользователь может явно вернуться к локальному `%AppData%\ScreensView\computers.json`.
9. При ошибке открытия или переключения текущий источник подключений и список компьютеров остаются без изменений.
10. `README.md` обновлён и описывает новый режим работы с внешним зашифрованным файлом подключений.
