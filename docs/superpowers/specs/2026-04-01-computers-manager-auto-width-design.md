# Автоматическая ширина окна управления компьютерами — дизайн-документ

**Дата:** 2026-04-01
**Проект:** ScreensView.Viewer (WPF, net8.0-windows)

---

## Контекст

Сейчас `ComputersManagerWindow` открывается с заранее заданной шириной.
После добавления новых действий в toolbar ширина строки кнопок выросла, и окно стало зависеть от вручную подобранного `Width` / `MinWidth`.

Это решение хрупкое:

- новый текст кнопки снова может сделать стартовую ширину недостаточной
- поведение зависит от DPI и системных шрифтов
- ширина окна задаётся не по фактическому содержимому, а по заранее выбранной константе

При этом текст кнопок в процессе работы пока не меняется, поэтому живое автоизменение размера окна не требуется.

---

## Решение

Выбран вариант с **однократным вычислением ширины при открытии окна**.

`ComputersManagerWindow` после первого layout-pass измеряет фактическую ширину верхнего toolbar и на основе этого выставляет стартовую ширину окна.
Получившееся значение применяется одновременно к `Width` и `MinWidth`.

Итоговая ширина ограничивается через `SystemParameters.WorkArea`, чтобы окно не выходило за границы доступной рабочей области экрана.

### Почему не фиксированная ширина

Фиксированное значение в XAML не отражает реальный размер toolbar и требует ручной подгонки после каждого изменения текста или набора кнопок.

### Почему не динамический resize во время работы

Тексты кнопок сейчас статичны.
Постоянное автоизменение размера окна в ответ на любые layout-изменения добавило бы лишнюю сложность и могло бы визуально раздражать пользователя.

---

## Поведение

### При открытии окна

1. Окно создаётся как обычно.
2. После завершения первого layout-pass код вызывает измерение toolbar.
3. Вычисляется требуемая ширина окна:
   - фактическая ширина toolbar-кнопок
   - плюс горизонтальные отступы контейнера
   - плюс системная ширина рамки окна / non-client area
4. Итог ограничивается значением `SystemParameters.WorkArea.Width`.
5. Полученное значение присваивается:
   - `Width`
   - `MinWidth`

### После открытия окна

- окно не продолжает автоматически менять размер
- пользователь по-прежнему может увеличивать окно
- пользователь не может сузить окно меньше вычисленного `MinWidth`

### Ограничения

Если требуемая ширина toolbar больше ширины рабочей области экрана, окно открывается в максимально допустимой ширине рабочей области.
В этом случае дальнейшее решение overflow-контента в toolbar не входит в эту задачу.

## Реализация

### XAML

В `ComputersManagerWindow.xaml`:

- убрать жёстко зафиксированный `MinWidth`, подобранный вручную
- оставить обычную стартовую ширину по умолчанию
- дать имя контейнеру toolbar-кнопок, чтобы его можно было измерить после layout

### Code-behind

В `ComputersManagerWindow.xaml.cs`:

- подписаться на одноразовый запуск логики после `ContentRendered`
- вызвать `UpdateLayout()`
- измерить фактическую ширину toolbar-контейнера
- вычислить итоговую ширину окна с учётом рамки и отступов
- ограничить её по `SystemParameters.WorkArea.Width`
- применить итог к `Width` и `MinWidth`

### Выделение helper-логики

Само вычисление целевой ширины лучше вынести в небольшой `internal` helper-метод или helper-класс, чтобы его можно было тестировать без UI-автоматизации и без создания полноценного окна WPF в тесте.

## Проверка

1. `ComputersManagerWindow` открывается уже достаточной ширины, чтобы toolbar помещался без ручного растягивания.
2. `MinWidth` после открытия равен вычисленному значению, а не жёсткой константе.
3. Если вычисленная ширина больше рабочей области экрана, окно ограничивается `SystemParameters.WorkArea`.
4. Окно не “прыгает” и не продолжает менять размер во время дальнейшей работы.
5. `dotnet test ScreensView.slnx` проходит без регрессий.
